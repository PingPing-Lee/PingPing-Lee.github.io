# 概述

传统开发模式的经常会面临一些问题，如变量命名冲突、变量全局污染、文件依赖管理混乱等，因此需要通过 通过模块化 来抽离公共代码、隔离作用域从而解决上述问题

模块化： 就是把单独的一个功能封装到一个模块中，模块之间相互隔离，但是可以通过特定接口公开内部成员，也可以依赖别的模块。
好处： 方便了代码的重用，从而提高开发效率、降低维护成本。

在 JS 中一开始本身没有提供模块化的机制，所以才会衍生出 commonJS、 AMD、CMD 这么多模块化规范。然后 JS 在ES6时也提供了原生的模块化机制，下面就来一一对比一下吧：

# CommonJS

CommonJS 并不是一个模块规范的名称，而是一个有志于构建 JavaScript 生态圈的组织，在他们意识到 JavaScript 中没有模块化机制带来的一系列问题之后，在 JavaScript 社区发起创造了一套模块系统的规范，并在 Node.js 上应用并推广，后续也影响到了浏览器端的 JavaScript

CommonJS 实现了服务器端(node) 和 浏览器端(webpack) 的模块化规范，每一个文件即一个模块，文件内部通过 module 变量来表示，用 module.exports 来导出自身，用 require 来引用加载其他依赖模块，需要特别注意模块加载方式在服务端和浏览器端是不同的，服务端是运行时同步加载，在浏览器端是提前编译打包处理，

示例引入方式如下：

// 引入 第三方
var fs = require('fs');
// 引入自定义模块
require('./xxx')
// 其实此时是加载 xxx 模块的 module.exports 属性，所以 xxx.js 的定义是
module.exports = { ... }

// 如果定义如下
exports.xxx = { name: 'module1' }
// 在引入使用时需要注意加上 .xxx，如下：
const module1 = require('module1');
console.log(module1) // { xxx: { name: 'module1' } }
console.log(module1.xxx) // { name: 'module1' }

# AMD

AMD 实现了浏览器端的模块化规范，典型代表有 Require.js (opens new window)，最大的特点是依赖前置
即通过 require 引入时，它会先加载对应的依赖，等依赖资源加载完之后，会执行回调函数，将依赖作为入参，执行对应的业务逻辑，示例引入方式如下：

require(['module1', 'module2'], function(m1, m2){
    // ... 使用 m1/m2
})

基本语法示例：

// 无依赖
define(function(){
    return 模块
})
// 有依赖
define(['module1', 'module2'], function(m1, m2){
    return 模块
})

特点：

  • 依赖模块必须提前声明好
  • 依赖模块的加载是异步的，指定回调函数

# CMD

CMD 和 AMD 一样也实现了浏览器端的模块化规范，典型代表有 Sea.js (opens new window)，强调强调延迟加载，对应的依赖在回调函数里执行具体依赖语句时，才会去加载，不过 AMD 在后续版本里也支持了延迟加载的写法

基本语法示例：

// 无依赖
define(function(require, exports,module){
    exports.xxx = value;
    module.exports = value
})
// 有依赖
define(function(require, exports,module){
    // 同步引入依赖模块
    var module1 = require('module1');
    // 同步引入依赖模块
    require.async('module1', function(m1){})
    exports.xxx = value;
    module.exports = value
})

特点：

  • 整合了 CommonJS 和 AMD 的特点
  • 依赖模块在使用时才声明
  • 依赖模块的加载是异步的，使用时才会加载执行

# ES6 Module

Javascript 社区虽尝试提出 AMD、CMD、和 CommonJS 等模块化规范，但是由于自身存在一定的差异性与局限性，因此定义了 ES6 模块化规范，是浏览器端和服务端通用的模块化开发规范标准, 不过在浏览器端需要用 babel 将 ES6 转换成 ES5

定义：

  • 每一个 JS 文件都是一个独立的模块
  • 使用 import 关键字 引入依赖模块
  • 使用 export 关键字 暴露依赖模块

默认导入导出：

// 当前模块为 export1.js 定义私有成员 a 和 c
let a = 10, b = 20;
function show(){};
// 外界访问不到变量 c 因为其没有被暴露出去
let c = 30;
// 通过 export default 将本模块中的私有成员暴露出去，供其他模块使用
export default  { a, b, show };

// import 默认导入 export1
import Export1 from './export1.js'
console.log(Export1); // { a: 10, b: 20, show: [function show] }

特点：

  • 在一个模块中只允许使用唯一的一次 export default，多次使用会报错
  • 在一个模块中没有向外 export default，则导入该模块时默认输出 {}
  • 在 import 导入模块加载时，不允许在引用变量，必须为真实的文件路径
  • ES6的模块机制在依赖模块时并不会先去预加载整个脚本，而是生成一个只读引用，并且静态解析依赖，等到执行代码时，再去依赖里取出实际需要的模块

与 CommonJS 的区别：

CommonJS ES6 Module
加载方式 Common 运行时加载 ES6编译时输出接口()
加载方式描述 CommonJS 加载的 module.exports 属性是一个对象，该对象只有在脚本运行完才会生成 ES6 加载的模块不是对象，只是一种静态定义，在代码静态解析阶段就会生成
输出方式 输出的是一个值的拷贝 输出的是一个值的引用
输出方式描述 模块可以多次加载，但是只会在第一次加载时运行一次，然后运行结果就会被缓存下来，后面再次加载就直接读取缓存结果，要想让模块再次运行，必须清除缓存 ES6模块是动态引用的，并不会缓存值，模块里面的变量绑定其所在的模块
