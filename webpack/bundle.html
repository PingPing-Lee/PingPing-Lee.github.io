# 打包构建

在 webpack 中通过以下命令行可以进行打包构建：

  • 通过 npm scripts 运行 webpack
    • 开发环境： npm run dev
    • 生产环境：npm run build
  • 通过 webpack 直接运行
    • webpack entry.js bundle.js

那么这个过程发生了什么呢

# bin\webpack.js

在命令行运行以上命令后，npm 会让命令行工具进入 node_modules.bin 目录查找是否存在 webpack.sh 或者 webpack.cmd 文件，如果存在，就执行，不存在，就抛出错误

在源码中查找会实际的入口文件是：node_modules\webpack\bin\webpack.js，仔细研究源码可以大致分为以下几个步骤：

    // 1. 正常执行默认返回 0，如果有错误会修改 exitCode 值
    process.exitCode = 0; 
    // 2. 运行某个命令 
    const runCommand = (command, args) =>{ ... };
    // 3. 判断某个包是否安装 
    const isInstalled = packageName =>{ ... };
    // 4. webpack 可用的 CLI: webpack-cli 和 webpack-command
    const CLIs =[...]; 
    // 5. 判断是否两个 ClI 是否安装了 
    const installedClis = CLIs.filter(cli => cli.installed);
    // 6. 根据安装数量进行处理 
    // 如果一个 ClI 都没有，询问是否进行安装，yes 时通过 runCommand 进行安装 webpack-cli
    if (installedClis.length === 0){
        ...
    }else if (installedClis.length === 1){ // 如果有一个 ClI 则执行 CLI
        ...
    }else { // 如果有俩个 ClI 则提示只能有一个CLI，请进行删除，并抛出错误 exitCode = 1
        ...
    }

其实通过上面源码的分析可以看出来 webpack 最终找到 webpack-cli (webpack-command) 这个 npm 包，并且 执行 CLI，那么 webpack-cli 具体做了什么事情呢？

# webpack-cli

webpack-cli 在源码中路径位置是 /node_modules/webpack-cli/bin/cli.js，源码：

// 处理不需要经过编译的命令，utils/constants 中定义了不需要编译的命令行数组
const { NON_COMPILATION_ARGS } = require("./utils/constants");
(function() {
    // 导入本地的包
    const importLocal = require("import-local");
    // 预编译本地包
    if (importLocal(__filename)) {
        return;
    }
    // V8 缓存
    require("v8-compile-cache");
    // 错误处理
    const ErrorHelpers = require("./utils/errorHelpers");
    // 如果有不需要编译的命令则走到 ./utils/prompt-command 进一步判断是否需要安装
    const NON_COMPILATION_CMD = process.argv.find(arg => { ... });
    if (NON_COMPILATION_CMD) {
        return require("./utils/prompt-command")(NON_COMPILATION_CMD, ...process.argv);
    }
    // 命令行工具包 yargs，对命令行进行定制
    const yargs = require("yargs").usage(`...`);
    // ./config/config-yargs 中对命令行进行分组 
    require("./config/config-yargs")(yargs);
    // 对命令行进行解析，对各个参数进行转换，组成编译配置项
    yargs.parse(process.argv.slice(2), (err, argv, output) => {
        ...
        let options;
        try {
            // ./utils/convert-argv 根据命令行或者 webpack 配置的参数给 options 去增加相应的参数和插件
            options = require("./utils/convert-argv")(argv);
        } catch (err) {
            ...
        }
    })
    ...
    function processOptions(options) {
        ...
        // 引入 webpack，根据配置项进行编译和构建
        const webpack = require("webpack");
        let compiler;
        try {
            compiler = webpack(options);
        } catch (err) {
            ...
        }
        ...
    }
})

  • 上述代码提到的 utils/constants 中定义的不需要编译的命令如下：

    //  utils/constants.js
const NON_COMPILATION_ARGS = [
    "init", // 创建一份 webpack 配置文件
    "migrate", // 进行 webpack 版本迁移 
    "add", // 往 webpack 配置文件中增加属 性 
    "remove", // 往 webpack 配置文件中删除属 性 
    "serve", // 运行 webpack-serve 
    "generate-loader", // 生成 webpack loader 代码 
    "generate-plugin", // 生成 webpack plugin 代码 
    "info", // 返回与本地环境相关的一些信息 
]

  • 上述代码中引入 yargs 命令行工具包，提供命令和分组参数，参数分组 (config/config-args.js)，将命令划分为9类：：

    • Config options: 配置相关参数(文件名称、运行环境等)
    • Basic options: 基础参数(entry设置、debug模式设置、watch监听设置、devtool设置)
    • Module options: 模块参数，给 loader 设置扩展
    • Output options: 输出参数(输出路径、输出文件名称)
    • Advanced options: 高级用法(记录设置、缓存设置、监听频率、bail等)
    • Resolving options: 解析参数(alias 和 解析的文件后缀设置)
    • Optimizing options: 优化参数
    • Stats options: 统计参数
    • options: 通用参数(帮助命令、版本信息等)

总结下来 webpack-cli 首先对配置文件和命令行参数进行转换最终生成配置选项参数 options，然后引入 webpack，根据配置参数实例化 webpack 对象，最后执行构建流程

# Tapable

上述代码中我们提到了 compiler，其实 compiler 和 Compilation 都是 webpack 的核心对象，但是查找源码会发现这两个都是继承 Tapable，如下：

class Compiler extends Tapable {...}
class Compilation extends Tapable {...}

那么 Tapable 是什么呢

Tapable 是一个类似于 Node.js 的 EventEmitter 的库, 主要是控制钩子函数的发布 与订阅,控制着 webpack 的插件系统

Tapable 库暴露了很多 Hook（钩子）类，为插件提供挂载的钩子

const {
    SyncHook, // 同步钩子
    SyncBailHook, // 同步熔断钩子
    SyncWaterfallHook, // 同步流水钩子
    SyncLoopHook, // 同步循环钩子
    AsyncParallelHook, // 异步并发钩子
    AsyncParallelBailHook, // 异步并发熔断钩子
    AsyncSeriesHook, // 异步串行钩子
    AsyncSeriesBailHook, // 异步串行熔断钩子
    AsyncSeriesWaterfallHook // 异步串行流水钩子 
} = require("tapable");

Tapable 暴露出来的都是类方法，new 一个类方法获得我们需要的钩子，如下是一个异步绑定钩子的方法：

// 类接受数组参数 options ，非必传。类方法会根据传参，接受同样数量的参数
const hook1 = new AsyncSeriesHook(["arg1", "arg2", "arg3"]); 
// 绑定一个异步 Promise 钩子到 webapck 事件流
hook1.tapPromise("calculateRoutes tapPromise", (arg1, arg2, arg3, callback) => {
    // return a promise
    return new Promise((resolve,reject)=>{
        setTimeout(()=>{
            console.log(`tapPromise to ${arg1} ${arg2} ${arg3}`)
            resolve();
        },1000)
    })
});
// 执行绑定的事件，入参数量最多只能是实例化时数组参数的长度
hook1.call(1,2,3)
//执行异步钩子
hook1.promise('Async', 'hook', 'demo').then(() => {
    console.timeEnd('cost');
}, err => {
    console.error(err);
    console.timeEnd('cost');
});

了解了 Tapable 的基础使用之后，是如何和 webpack 联系起来的呢

let compiler;
if (Array.isArray(options)) {
    compiler = new MultiCompiler(
        Array.from(options).map(options => webpack(options))
    );
} else if (typeof options === "object") {
    options = new WebpackOptionsDefaulter().process(options);

    compiler = new Compiler(options.context);
    compiler.options = options;
    // 插件 plugin 必须有 apply 方法
    new NodeEnvironmentPlugin({
        infrastructureLogging: options.infrastructureLogging
    }).apply(compiler);
    if (options.plugins && Array.isArray(options.plugins)) {
        for (const plugin of options.plugins) {
            if (typeof plugin === "function") {
                plugin.call(compiler, compiler);
            } else {
                plugin.apply(compiler);
            }
        }
    }
    compiler.hooks.environment.call();
    compiler.hooks.afterEnvironment.call();
    // 注入内部插件
    compiler.options = new WebpackOptionsApply().process(options, compiler);
}
  • 插件需要有 apply 方法接受 compiler 参数
  • 插件上做一些事件的监听 compiler 中的 hooks 里关键的事件节点 判断 options 是否哟 plugins，如果有通过遍历组装参数

# 源码学习

通过对源码的构建流程的学习，可以将 webpack 的编译都按照下面的钩子调用顺序执行：

整体可以分为四个阶段，准备阶段、make 构建阶段、seal 构建优化阶段、emit 阶段，具体分析：

  • 准备阶段
    • 主要是通过 WebpackOptionsApply 将所有的配置 options 参数转换成 webpack 内部插件
    • 如通过 WebpackOptionsApply 中 EntryOptionPlugin 判断 entry 的类型是字符串、数组还是function 然后转换成相应的插件
    • 下面列举了一些参数默认被转换成插件的对应列表：
      • output.library -> LibraryTemplatePlugin
      • externals -> ExternalsPlugin
      • devtool -> EvalDevtoolModulePlugin, SourceMapDevToolPlugin
      • AMDPlugin, CommonJsPlugin
      • RemoveEmptyChunksPlugin
  • make 构建阶段
    • 初始化生成的内部插件都会通过 apply(compiler) 实例化一个 compiler 对象，对象中有 compilation、NormalModuleFactory、contextModuleFactory，以及执行 run 函数，
    • 在 run 函数执行之前会先执行 beforeRun 钩子去删除一些缓存，然后再执行 run 钩子进行编译，
    • 编译是通过 compile 函数，同样先执行 beforeCompile 钩子，然后触发 make 钩子，
    • 监听 make 钩子的插件，通过 compilation.addEntry 添加入口文件，
    • addEntry 中的 addModuleChain 将模块添加到依赖列表中，同时通过 buildModule 进行模块的构建，
    • 在 buildModule 中的 doBuild 方法通过触发 runLoaders 将代码进行 Loader 转换，
    • doBuild 之后触发 parser.parse 分析模块的依赖，并且进行构建，
    • 构建完成之后会触发 compilation.finish 函数，得到经过 Loaders 处理之后的源码，至此 make 阶段完成
  • seal 构建优化阶段
    • make 阶段完成，会走到 seal 阶段，在 beforeChunks 之前做优化工作，同时创建 Hash，然后 createModuleAssets 生成内容，将 Chunk 添加到 compilation.assets 中，至此整个 seal 阶段完成
    • 这里Chunk 生成算法
      • webpack 先将 entry 中对应的 module 都生成一个新的 chunk
      • 遍历 module 的依赖列表，将依赖的 module 也加入到 chunk 中
      • 如果一个依赖 module 是动态引入的模块，那么就会根据这个 module 创建一个 新的 chunk，继续遍历依赖
      • 重复上面的过程，直至得到所有的 chunks
  • emit 阶段
    • 通过 compilation.getPath 获取输出到内容，然后通过 outputFileSystem.mkdirp 将输出的内容存到磁盘
